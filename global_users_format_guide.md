# SOYAL Proxy - 全域白名單 (global_users.json) 設定指南

當您執行 `-sync-down-all` 或是準備上傳全域白名單給設備時，程式會讀取或覆寫 `global_users.json` 檔案。
這份文件專門用來解釋 JSON 的架構，以及各個可選進階設定欄位的規則與運作方式。

---

## 一、 基礎架構概述
整個 JSON 檔案是一個陣列 (Array)，裡面包含多個「以人/卡號為單位」的物件。
如果您有一張卡號要同時開通多台設備的權限，只要把它們包在 `permissions` 底下對應的 `Node ID` 區塊即可。

### 最精簡的範例 (僅刷卡開門)
如果您什麼進階功能都不用，請維持這種最乾淨的格式即可：
```json
[
  {
    "card_id": "01234:56789",
    "notes": "業務部 王大明",
    "permissions": {
      "1": {},
      "3": {}
    }
  }
]
```
> **程式防呆機制**：
> 當您省略 `user_addr` 與所有進階參數時，一旦執行 `-sync-up-all`，程式會「**自動尋找空位配發**」，並主動注入「**無密碼、期限至 2099 年、開全區全群組、純刷卡模式**」的預設設定給 Node 1 和 Node 3 設備，保證能順利感應開門。

---

## 二、 完整欄位定義與可選內容 (Optional Fields)

在 `permissions` 底下的每個節點 (例如 `"1"`、`"3"`)，皆可以自由加上下列「**進階選項**」：

| 欄位名稱 (JSON) | 型別 | 說明 / 預設值 |
|---|---|---|
| `user_addr` | 數字 | **設備記憶體內的會員位址 (0 ~ 16383)**。<br>💡【小撇步】：如果您手動建立新卡但不知道機器還剩幾號空位，**請完全刪除此欄位**。程式在更新時會自動幫它指派最新的空號並回寫到這份表中。 |
| `pin` | 字串 | **卡加自訂密碼模式的 PIN 碼** (需搭配修改 `mode` 屬性)。<br>💡 預設：空字串 (不需密碼)。<br>💡 範例：`"1234"`、`"5678"`。 |
| `expiry` | 字串 | **卡片的使用期限 (到期日)**。<br>💡 格式：`YYYY-MM-DD` (例如："2026-12-31")<br>💡 預設：當您未填寫時，程式自動發放至 `"2099-12-31"` 永久有效。 |
| `floors` | 數字陣列 | **電梯控制器的樓層放行陣列** (限設備為包含樓層機的控制器時有效)。<br>💡 填寫此卡能感應抵達的樓層編號。<br>💡 範例：`[1, 2, 3, 10, 15]` (開通 1,2,3,10,15樓)。 |
| `mode` | 字串 | **門禁通行模式**。<br>💡 預設：`"card"` (純刷卡)。<br>💡 選項：若要改為密碼模式，可填寫 `"card_or_pin"` (卡或密碼) 或是 `"card_and_pin"` (刷卡加密碼)。 |
| `zone` | 數字 | **時段管制 (Time Zone)**。<br>💡 預設：`0` (不限時段、自由通行)。 |
| `doors` | 數字陣列 | **大門群組權限陣列 (對應 RS-485 副讀卡機 1~16)**。<br>💡 預設：未填寫時為**開放所有大門**。<br>💡 範例：`[1, 2, 3]` (只開通第 1, 2, 3 號副讀卡機門鎖)。 |

## 三、 進階操作範例

### 範例 A：電梯限制特定樓層
這位員工只有 1 號大門前門以及 3 號電梯 (1, 2, 10 樓) 的權限：
```json
[
  {
    "card_id": "01234:56789",
    "notes": "研發部 陳小美",
    "permissions": {
      "1": {},
      "3": {
        "floors": [1, 2, 10]
      }
    }
  }
]
```

### 範例 B：外包人員限制使用期限
針對短期施工的外包駐點人員，只開通 Node 2 的後門權限，且限定於一個月後過期：
```json
[
  {
    "card_id": "99999:88888",
    "notes": "外包施工 李師傅",
    "permissions": {
      "2": {
        "expiry": "2024-05-31"
      }
    }
  }
]
```

### 範例 C：使用密碼 + 手動指定位址
如果在特定的高風險機房 (Node 4)，員工除了刷卡還必須輸入專用的獨立 PIN 碼，且您想要手動將這位員工強制定死在設備第 `50` 號位址：
```json
[
  {
    "card_id": "11111:22222",
    "notes": "機房管理員",
    "permissions": {
      "4": {
        "user_addr": 50,
        "pin": "1846",
        "mode": "card_or_pin"
      }
    }
  }
]
```

## 常見問題與管理技巧

1. **如果在 JSON 中刪除某人的 Node Block (例如刪掉 `"3": {}`)，他還能進門嗎？**
   CLI 這支工具主要是以**「更新 / 指派 (Sync Up)」**為導向的，它會把您 JSON 裡包含的卡片對應寫進該節點內。
   如果您要徹底刪掉某個人在機器上的權限，您可能需要額外的「刪除卡號」程式邏輯 (呼叫 `85H` 指令)。目前最快的替代作法是：將他的 `"card_id"` 改成一組空亂碼，再次 `sync-up-all` 覆蓋掉他本來那個 `user_addr` 的位址。

2. **為什麼有時候 `-sync-down-all` 出來的 JSON 會有 `"doors"` 等欄位？**
   因為 `soyal_proxy.exe` 在從設備逆向讀出資料時，只要發現那台設備內的記憶體設定值與預設值不同（或確實有資料），它就會誠實地把它反序列化呈現出來。不影響您下次上傳，您若覺得繁雜也能直接手動在編輯器把那些欄位刪掉，程式上傳時會幫您以預設寬鬆值補齊。
